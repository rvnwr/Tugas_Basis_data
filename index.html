1) Harapan dari Mata Kuliah Basis Data

Dari mata kuliah ini, saya berharap tidak hanya memahami teori, tetapi juga:

  • Mampu merancang database dari nol sesuai kebutuhan aplikasi.
  • Bisa menggunakan SQL dengan percaya diri untuk query data.
  • Memahami bagaimana keamanan dan integritas data dijaga.
  • Menghubungkan konsep database dengan dunia nyata, misalnya sistem informasi kampus.

2) Apa itu Basis Data

Basis data adalah tempat penyimpanan informasi yang terstruktur, sehingga bisa diakses, diubah, dan dianalisis dengan cepat. Sehari-hari kita berinteraksi dengan basis data tanpa sadar, misalnya:

  • Aplikasi e-commerce yang menyimpan data produk & transaksi.
  • Media sosial yang menyimpan postingan, komentar, dan like.
  • Sistem akademik yang menyimpan data mahasiswa, mata kuliah, dan nilai.

3) Apa itu Conceptual Database

Conceptual Database adalah tahap awal dalam perancangan database, yang fokus pada gambaran umum tanpa detail teknis.

Biasanya divisualisasikan dengan Entity-Relationship Diagram (ERD) yang menunjukkan:

  • Entitas (contoh: Mahasiswa, Dosen, Mata Kuliah).
  • Atribut (contoh: Nama, NIM, Alamat).
  • Relasi antar entitas (contoh: Mahasiswa mengambil Mata Kuliah).

4) Apa itu Physical Model

Physical Model adalah implementasi nyata dalam sistem DBMS, di mana desain konseptual diwujudkan menjadi tabel, kolom, dan tipe data.

Contoh sederhana dalam SQL: 

CREATE TABLE Mahasiswa (
    NIM VARCHAR(10) PRIMARY KEY,
    Nama VARCHAR(50),
    Prodi VARCHAR(30),
    Angkatan INT
);

Pada tahap ini juga dipikirkan performa, keamanan, serta strategi backup data.

Penjelasan Entity-Relationship Diagram (ERD)

1. Entitas

Entitas adalah objek nyata atau konsep yang datanya ingin disimpan dalam database. Contoh entitas: Mahasiswa, Dosen, Mata Kuliah.

2. Atribut

Atribut adalah informasi yang dimiliki oleh entitas. Contoh: Mahasiswa memiliki NIM, Nama, Alamat, Tanggal Lahir.

3. Relasi

Relasi menunjukkan hubungan antar entitas. Contoh: Mahasiswa mengambil Mata Kuliah, Dosen mengajar Mata Kuliah. Relasi bisa memiliki atribut tambahan, misal Nilai pada relasi Mengambil.

4. Kardinalitas

Kardinalitas menjelaskan jumlah hubungan antar entitas. Contoh: 1 Mahasiswa bisa mengambil banyak Mata Kuliah (1:N), 1 Dosen bisa mengajar banyak Mata Kuliah (1:N), Mata Kuliah bisa diambil banyak Mahasiswa (M:N).

5. Contoh Implementasi SQL

CREATE TABLE Mahasiswa (
    NIM VARCHAR(10) PRIMARY KEY,
    Nama VARCHAR(50),
    Alamat VARCHAR(100),
    TglLahir DATE
);

CREATE TABLE Dosen (
    NIP VARCHAR(10) PRIMARY KEY,
    Nama VARCHAR(50),
    Departemen VARCHAR(50)
);

CREATE TABLE MataKuliah (
    KodeMK VARCHAR(10) PRIMARY KEY,
    NamaMK VARCHAR(50),
    SKS INT,
    NIP_Dosen VARCHAR(10),
    FOREIGN KEY (NIP_Dosen) REFERENCES Dosen(NIP)
);

CREATE TABLE Mengambil (
    NIM VARCHAR(10),
    KodeMK VARCHAR(10),
    Nilai CHAR(2),
    PRIMARY KEY (NIM, KodeMK),
    FOREIGN KEY (NIM) REFERENCES Mahasiswa(NIM),
    FOREIGN KEY (KodeMK) REFERENCES MataKuliah(KodeMK)
);

Primary Key, Foreign Key, Candidate Key

1) Apa itu Primary Key, Foreign Key, dan Candidate Key?

Dalam perancangan basis data, key berfungsi untuk memastikan keunikan data dan hubungan antar tabel.

‚Ä¢ Primary Key (Kunci Utama)

Primary Key adalah atribut yang secara unik mengidentifikasi setiap baris (record) dalam tabel. Tidak boleh bernilai null dan tidak boleh duplikat.

  • NIM pada tabel Mahasiswa
  • NIP pada tabel Dosen
  • KodeMK pada tabel MataKuliah
  • ID_Pesanan pada tabel Penjualan
  • ID_Karyawan pada tabel Karyawan

‚Ä¢ Foreign Key (Kunci Tamu)

Foreign Key digunakan untuk menghubungkan satu tabel dengan tabel lain. Atribut ini merupakan kunci utama dari tabel lain yang menjadi referensinya.

  • NIP_Dosen pada tabel MataKuliah (mengacu ke Dosen)
  • NIM pada tabel Mengambil (mengacu ke Mahasiswa)
  • KodeMK pada tabel Mengambil (mengacu ke MataKuliah)
  • ID_Pelanggan pada tabel Penjualan (mengacu ke Pelanggan)
  • ID_Karyawan pada tabel Absensi (mengacu ke Karyawan)

‚Ä¢ Candidate Key (Kunci Kandidat)

Candidate Key adalah atribut atau kombinasi atribut yang berpotensi menjadi Primary Key karena bersifat unik. Dari beberapa Candidate Key, satu akan dipilih sebagai Primary Key.

  • NIM dan Email pada tabel Mahasiswa
  • NIP dan NomorHP pada tabel Dosen
  • KodeMK dan NamaMK pada tabel MataKuliah
  • ID_Pesanan dan NomorNota pada tabel Penjualan
  • ID_Barang dan Barcode pada tabel Inventori

Penjelasan Relation Pada ERD

2Ô∏è) Apa itu Relasi pada ERD?

Relasi adalah penghubung antar entitas dalam sistem. Relasi dapat menggambarkan bagaimana dua atau lebih entitas saling berhubungan. Terdapat tiga jenis utama relasi:

  • One to One (1:1) ‚Üí Satu entitas hanya berhubungan dengan satu entitas lain.
  • One to Many (1:N) ‚Üí Satu entitas berhubungan dengan banyak entitas lain.
  • Many to Many (M:N) ‚Üí Banyak entitas dari satu tabel berhubungan dengan banyak entitas dari tabel lain.

Kenapa Many to Many Tidak Baik Digunakan Langsung?

Relasi Many-to-Many tidak direkomendasikan langsung dalam database relasional karena sulit dikelola dan tidak efisien saat di-query. Solusinya adalah membuat tabel penghubung (junction table) agar relasi menjadi dua relasi One-to-Many. 

Contoh:
Mahasiswa (NIM, Nama)
MataKuliah (KodeMK, NamaMK)
Mengambil (NIM, KodeMK, Nilai)

Apa itu Normalisasi?

Normalisasi adalah proses mengatur data dalam database agar tidak terjadi redundansi (pengulangan data) dan memastikan integritas data terjaga. Dalam pembuatan ERD, normalisasi membantu agar setiap tabel memiliki struktur yang efisien dan mudah diubah di masa depan.

Tahapan Normalisasi

  • 1NF (First Normal Form) ‚Äì Setiap kolom hanya berisi nilai tunggal.
  • 2NF (Second Normal Form) ‚Äì Sudah memenuhi 1NF dan semua atribut bergantung penuh pada primary key.
  • 3NF (Third Normal Form) ‚Äì Sudah memenuhi 2NF dan tidak ada ketergantungan antar atribut non-key.

Contoh Sederhana:

Sebelum Normalisasi:
Tabel Mahasiswa (NIM, Nama, KodeMK, NamaMK, Dosen)

Setelah Normalisasi:

  • Tabel Mahasiswa (NIM, Nama)
  • Tabel MataKuliah (KodeMK, NamaMK, Dosen)
  • Tabel KRS (NIM, KodeMK)
